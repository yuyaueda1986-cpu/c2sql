# libc2sql ユーザーストーリー

本書は libc2sql を「**管理者がバッチ作業でC言語ユーザにDBアクセスライブラリを提供する**」という運用モデルで捉え直したときの、登場人物・作業の流れ・受け渡し様式を定義する。実装現状とのギャップおよび改善ロードマップは [改善提案.md](改善提案.md) を参照。

---

## 1. 登場人物（ペルソナ）

| 役割 | 別名 | 関心事 | 本書での略称 |
|--|--|--|--|
| **C言語ユーザ** | 利用者・アプリ開発者 | SQLを書かず、自分の構造体をそのまま永続化したい | 利用者 |
| **DB管理者** | 管理者・ライブラリ提供者 | 利用者からの依頼を**手作業ゼロ**で定義化し、ビルド済みライブラリを払い出したい | 管理者 |
| **運用者** | DBメンテナ | 払い出し後のDBのバックアップ・マイグレーション・整合性監視 | 運用者 |

- 利用者と管理者は**別の人物・別の責務**である点が本モデルの肝。利用者はSQLもスキーマDDLも知らなくてよい。
- 管理者の作業は属人的な手作業ではなく、**bash / Python / C製ツールによるバッチ**で再現可能にする。

---

## 2. ストーリー全体像

```
┌──────────┐   ①スキーマ仕様(JSON/YAML)   ┌──────────┐
│  利用者   │ ───────────────────────────► │  管理者   │
│ (C開発者) │   構造体・主キー・最大件数   │(バッチ係) │
└────┬─────┘                              └────┬─────┘
     │                                          │ ②バッチ実行
     │                                          │   gen → build → test
     │   ④生成ヘッダ + libc2sql.a + DBファイル   ▼
     │ ◄──────────────────────────────── [c2sql-gen / CMake / ctest]
     ▼                                          │
┌──────────┐                                    │ ③成果物パッケージ
│ アプリ実装 │  #include "persons_schema.h"      │
│ (CRUD)    │  SqlRDBWrite(h,"persons",&p,NULL)  │
└──────────┘                                    ▼
                                          ┌──────────┐
                                          │  運用者   │ バックアップ/監視
                                          └──────────┘
```

## 3. ストーリー詳細

### ストーリーA — 利用者がDB情報を渡す

> 利用者である私は、永続化したい **C構造体・主キー・最大レコード数** を管理者に渡したい。
> そうすれば、SQLやスキーマDDLを自分で書かずにDBを使えるからだ。

利用者は「スキーマ仕様書」を1ファイル記入して管理者へ渡す。記入するのは次の情報のみ：

- 構造体（テーブル）名
- 各フィールドの「名前 / 型 / 主キーか / NULL許容か / UNIQUEか / TEXT/BLOBの長さ」
- **最大レコード数**（容量見積り・キャパシティ保護のため）
- 利用バックエンド（`sqlite3`、将来 `postgres`）

受け渡し様式（案）は [§5 受け渡し仕様](#5-受け渡し仕様スキーマ仕様書) を参照。

**受け入れ条件:**
- 利用者はC構造体のメモリレイアウト（`offsetof` / `sizeof`）を意識しなくてよい。
- 利用者はSQLite・DDL・マイグレーション手順を知らなくてよい。

---

### ストーリーB — 管理者が定義を生成しライブラリを払い出す（バッチ化）

> 管理者である私は、受け取った仕様書を**ワンコマンドのバッチ**で定義生成・ビルド・検証したい。
> そうすれば、依頼ごとの手作業とヒューマンエラー（オフセット・サイズの取り違え）を無くせるからだ。

管理者のバッチは次のステップを自動実行する：

1. **検証 (lint)** — 仕様書のスキーマ妥当性、識別子規約、予約語、型と長さの整合をチェック。
2. **定義生成 (gen)** — 仕様書から以下を生成：
   - `<name>_schema.h` … C構造体 `typedef` ＋ `extern const SqlRDBColumnDef[]` 宣言 ＋ 件数マクロ ＋ `最大レコード数` マクロ
   - `<name>_schema.c` … `offsetof` / `sizeof` で埋めた `SqlRDBColumnDef[]` 定義 ＋ 登録ヘルパ関数 `Register<Name>(SqlRDBHandle*)`
   - CMake 断片（生成物の組み込み）
3. **ビルド (build)** — `libc2sql.a` と生成定義をリンクしてスモークプログラムをコンパイル。
4. **検証 (test)** — `SqlRDBInit` → 生成 `Register<Name>()` が `SQL_RDB_OK` を返すこと、Write/Read のラウンドトリップが通ることを自動テスト。
5. **払い出し (package)** — 生成ヘッダ・`libc2sql.a`・空のDBファイル（スキーマ適用済み）・利用手順を1パッケージにまとめる。

**受け入れ条件:**
- 管理者の作業は `bash tools/build_batch.sh specs/*.json` 相当のワンコマンドで完了する。
- 手書きの `SqlRDBColumnDef[]` 配列は生成物に置き換わり、オフセット／サイズ不整合が構造的に発生しない。
- 1件でも検証に失敗した仕様は払い出しブロックされ、原因がログに出る。

---

### ストーリーC — 利用者がライブラリを使う

> 利用者である私は、払い出されたヘッダを include して構造体ポインタを渡すだけでCRUDしたい。

```c
#include "c2sql.h"
#include "persons_schema.h"   /* 管理者が生成・払い出し */

int main(void) {
    SqlRDBHandle *h = SqlRDBInit("app.db");
    RegisterPersons(h);              /* 生成された登録ヘルパ。手書き配列は不要 */

    Person alice = { 1, "Alice", 9.5 };
    SqlRDBWrite(h, "persons", &alice, NULL);

    SqlRDBClose(h);
    return 0;
}
```

**受け入れ条件:**
- 利用者コードにスキーマ定義（列名・型・オフセット）が現れない。
- 構造体の変更が必要になったら、仕様書を直して管理者に再依頼するだけでよい（§ストーリーD）。

---

### ストーリーD — スキーマ変更の再依頼

> 利用者である私は、フィールド追加が必要になったとき、仕様書を更新して再依頼したい。
> そうすれば、管理者のバッチが差分を判定し、安全なマイグレーション付きで再払い出ししてくれるからだ。

- 管理者バッチは**末尾カラム追加**を検出し、`auto_migrate` 前提の登録コードを生成する。
- カラム順序変更・型変更・カラム削除・主キー変更など `auto_migrate` で扱えない差分は、バッチが検出して**手動マイグレーションSQLの雛形**を出力し、運用者にエスカレーションする（[maintenance-guide.md](maintenance-guide.md) §5 参照）。

## 4. バッチ化のスコープ

| 対象 | バッチ化 | 担当ツール（案） |
|--|--|--|
| 仕様書の検証（lint） | ✅ | Python（`c2sql-lint`） |
| 定義生成（H/C/CMake） | ✅ | Python（`c2sql-gen`） |
| ビルド・テスト | ✅ | bash + CMake/ctest |
| 払い出しパッケージング | ✅ | bash |
| 最大レコード数の取り扱い | ⚠️ 要設計 | [改善提案.md §最大レコード数](改善提案.md) |
| 手動マイグレーション実行 | ❌（運用者の判断） | 雛形SQL生成までをバッチ化 |
| PostgreSQL バックエンド | 🔜 将来 | ドライバ抽象は実装済み |

## 5. 受け渡し仕様（スキーマ仕様書）

利用者が記入し管理者へ渡す、機械可読な1ファイル。**現状は未実装の提案フォーマット**（[改善提案.md](改善提案.md) で策定）。

```json
{
  "schema_version": "1.0",
  "struct_name": "persons",
  "backend": "sqlite3",
  "max_records": 100000,
  "fields": [
    { "name": "id",    "type": "int32", "primary_key": true },
    { "name": "name",  "type": "text",  "size": 32 },
    { "name": "score", "type": "real",  "nullable": true }
  ]
}
```

| キー | 必須 | 意味 | 対応する現行API要素 |
|--|--|--|--|
| `struct_name` | ✅ | テーブル／構造体名（識別子規約） | `SqlRDBRegisterStruct` の `struct_name` |
| `backend` | ✅ | `sqlite3`（将来 `postgres`） | ドライバ選択 |
| `max_records` | ✅ | 最大レコード数（容量保護／見積り） | **現状未対応**（要追加） |
| `fields[].name` | ✅ | カラム名 | `SqlRDBColumnDef.name` |
| `fields[].type` | ✅ | `int32`/`int64`/`real`/`text`/`blob` | `SqlRDBColumnDef.type` |
| `fields[].size` | text/blobで必須 | バッファ長 | `SqlRDBColumnDef.size` |
| `fields[].primary_key` | 任意 | 主キー（最大1） | `SQL_COL_FLAG_PRIMARY_KEY` |
| `fields[].nullable` | 任意 | NULL許容 | `SQL_COL_FLAG_NULLABLE` |
| `fields[].unique` | 任意 | UNIQUE制約 | `SQL_COL_FLAG_UNIQUE` |

> `offset` と `int32/int64/real` の `size` は生成器が `offsetof` / `sizeof` で機械的に補完するため、利用者は記入しない（手書き由来の不整合を排除）。

## 6. 現状とのギャップ（サマリ）

| 観点 | 現状 | ストーリーが求めるもの |
|--|--|--|
| 定義の作り方 | 利用者が `SqlRDBColumnDef[]` を手書き | 管理者バッチが仕様書から自動生成 |
| コード生成 | `design.md` で**非ゴール**と明記 | バッチ化の中核機能として**必要** |
| 最大レコード数 | 概念が存在しない | 受け渡し情報の一級項目 |
| 受け渡し様式 | 無し（口頭／コード共有） | 機械可読なスキーマ仕様書 |
| ツール群 | 無し（`tools/` ディレクトリ無し） | lint/gen/build のバッチ |
| 役割分担 | 利用者＝管理者（同一） | 利用者と管理者を分離 |

詳細なギャップ分析と段階的な改善ロードマップは **[改善提案.md](改善提案.md)** に記載する。
