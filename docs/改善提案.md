# libc2sql 改善提案 — 「管理者バッチによるライブラリ払い出し」モデルへの拡張

本書は [ユーザーストーリー.md](ユーザーストーリー.md) で定義した運用モデル（利用者が渡したDB情報を、管理者がバッチ作業で定義化しライブラリを払い出す）を実現するための**ギャップ分析**と**段階的改善ロードマップ**を策定する。

---

## 1. 現状サマリ

libc2sql v0.1.0 は完成度の高い**ランタイムライブラリ**である。

- 利用者が `SqlRDBColumnDef[]` 配列を**手書き**し、実行時に `SqlRDBRegisterStruct` で登録する。
- 全値プリペアドバインドによるSQLインジェクション排除、スレッドセーフ、ステートメントキャッシュ、`auto_migrate` などの品質機能を備える。
- ドライバ抽象（vtable）により PostgreSQL 等への拡張余地がある。

一方、ユーザーストーリーが前提とする「**利用者と管理者の分離**」「**管理者作業のバッチ化（コード生成）**」「**最大レコード数**」は、現設計の射程外にある。`docs/design.md` の非ゴールには「構造体定義からのコード生成（マクロ・ジェネレータ）」が明記されており、本提案はこの方針の見直しを含む。

---

## 2. ギャップ分析

| # | ギャップ | 現状 | 影響 | 重要度 |
|--|--|--|--|--|
| G1 | **コード生成が非ゴール** | `design.md` で明示的に除外 | バッチ化の中核が設計方針と矛盾 | 高 |
| G2 | **受け渡し様式が無い** | 口頭／ソース共有 | 利用者⇄管理者の契約が非形式・属人化 | 高 |
| G3 | **手書き `SqlRDBColumnDef[]`** | 利用者が `offset`/`size` を手入力 | オフセット・サイズ取り違えが既知の頻出ミス（api-guide §13） | 高 |
| G4 | **最大レコード数が無い** | 概念自体が不在 | 受け渡し情報の一級項目を保持・活用できない | 中 |
| G5 | **ツール群が無い** | `tools/` ディレクトリ無し | lint/gen/build/package を手作業で実施 | 高 |
| G6 | **役割分離が前提化されていない** | 利用者＝管理者を想定 | 払い出しパッケージの概念・手順が未定義 | 中 |
| G7 | **PostgreSQL は宣言のみ** | ドライバIFのみ、実装無し | `backend` 選択肢が将来限定 | 低（将来） |

---

## 3. 改善提案

### 提案1 — スキーマ仕様フォーマットの策定（G2 解消）

**目的:** 利用者⇄管理者の受け渡しを機械可読な契約にする。

**内容:**
- JSON を一次フォーマットとする（YAML はオプション）。スキーマは [ユーザーストーリー.md §5](ユーザーストーリー.md#5-受け渡し仕様スキーマ仕様書) の通り。
- JSON Schema（`tools/spec.schema.json`）でバリデーション可能にする。
- `offset` と数値型の `size` は**仕様書に書かせない**（生成器が補完）。これが G3 の根治。

**成果物:** `tools/spec.schema.json`、仕様書サンプル `specs/persons.json`。

---

### 提案2 — 定義生成ツール `c2sql-gen`（G1, G3 解消）

**目的:** 仕様書から、手書きを排した定義一式を自動生成する。

**生成物（`persons.json` → 例）:**

| ファイル | 内容 |
|--|--|
| `persons_schema.h` | `Person` 構造体 `typedef`、`extern const SqlRDBColumnDef PERSON_COLS[]`、`PERSON_COL_COUNT`、`PERSON_MAX_RECORDS` マクロ、`Register Persons()` 宣言 |
| `persons_schema.c` | `offsetof`/`sizeof` で埋めた `PERSON_COLS[]` 定義、`RegisterPersons(SqlRDBHandle*)` ヘルパ |
| `persons.cmake` | 生成物をターゲットに追加する CMake 断片 |

**設計上の判断（G1）:** `design.md` の非ゴール「コード生成」を改訂し、「**ライブラリ本体はコード生成に非依存。コード生成は外部ツール `tools/` として提供する**」と位置づける。ライブラリのABI・公開ヘッダは不変のまま、生成物はその利用者コードに過ぎないため、本体の単純さを損なわない。

**実装言語:** Python（標準ライブラリのみ、外部依存なし）。テンプレートは文字列ベースで十分。

**成果物:** `tools/c2sql_gen.py`、生成テンプレート、生成物のゴールデンテスト。

---

### 提案3 — 最大レコード数のサポート（G4 解消）

**目的:** 受け渡し情報 `max_records` を保持し、容量保護に活用する。現状ライブラリには対応概念が無いため、段階導入する。

**選択肢と評価:**

| 案 | 方式 | 長所 | 短所 | 採否 |
|--|--|--|--|--|
| A | 生成ヘッダの `#define <NAME>_MAX_RECORDS n` のみ（メタデータ） | 本体改修ゼロ・即導入可 | 強制力なし（見積り用途） | **Phase 1 採用** |
| B | 生成する登録ヘルパ内で、Write前に `COUNT(*)` 比較し超過を弾く薄いラッパ | アプリ改修不要・本体非改修 | カウントコスト／TOCTOU注意 | **Phase 2 採用（任意）** |
| C | SQLite トリガで件数上限を強制 | DB側で堅牢 | バックエンド依存・Postgres移植性 | 将来検討 |
| D | 本体に容量上限APIと `SQL_RDB_ERR_CAPACITY_EXCEEDED` を追加 | 一級機能化 | 公開ABI変更・スコープ増 | 必要時のみ |

**推奨:** まず案A（メタデータとして生成ヘッダに露出、容量見積り・ドキュメント用途）。強制が要件化したら案B（生成ラッパで `COUNT(*)` ガード）をオプトインで追加。案D（本体API化）はニーズが固まってから。

**成果物:** 生成ヘッダのマクロ、Phase 2 で `c2sql_capacity.h` 相当の生成ラッパ。

---

### 提案4 — バッチパイプライン `tools/`（G5 解消）

**目的:** 管理者作業をワンコマンド化する。

**ディレクトリ構成（案）:**

```
tools/
  spec.schema.json     # 仕様書のJSON Schema
  c2sql_lint.py        # 仕様書バリデーション
  c2sql_gen.py         # 仕様書 → H/C/CMake 生成
  build_batch.sh       # lint → gen → build → test → package を一括実行
  package.sh           # 払い出しパッケージング
specs/
  persons.json         # 受け渡し仕様書サンプル
```

**`build_batch.sh specs/*.json` の流れ:**

```
for spec in specs/*.json:
  c2sql_lint.py  <spec>            # 失敗したらブロック
  c2sql_gen.py   <spec> -o gen/    # 定義生成
build libc2sql.a + gen/*.c + smoke # CMake/ctest
package.sh gen/ build/ -> dist/<name>/   # ヘッダ+lib+空DB+手順
```

**成果物:** 上記スクリプト群、`dist/<name>/` 払い出しレイアウト定義。

---

### 提案5 — 払い出しパッケージの定義（G6 解消）

**目的:** 「利用者に何を渡すか」を標準化する。

**`dist/<name>/` の中身（案）:**
- `include/`（`c2sql.h` ＋ 生成 `<name>_schema.h`）
- `lib/libc2sql.a`
- `<name>.db`（スキーマ適用済みの空DB、任意）
- `README`（include/link 手順、`Register<Name>()` の呼び方）

---

### 提案6 — CI／検証フックの整備

**目的:** バッチ生成物の品質を機械保証する。

- 生成 → コンパイル → `Register<Name>()` が `SQL_RDB_OK` → Write/Read ラウンドトリップ、を ctest に追加。
- 生成物のゴールデンファイル比較（テンプレート退行検知）。
- 既存の ASan/UBSan・clang-tidy パイプラインに生成物を載せる。

## 4. ロードマップ（段階導入）

| フェーズ | 内容 | 提案 | 本体改修 | 状態 |
|--|--|--|--|--|
| **Phase 0** | ユーザーストーリー・改善提案のドキュメント化、`design.md` 非ゴールの見直し方針提示 | — | なし | ✅ 完了 |
| **Phase 1** | 仕様フォーマット＋ `c2sql-gen`（H/C/CMake生成）＋ `max_records` をメタデータ化＋ bash バッチ | 提案1,2,3A,4,5 | なし（外部ツールのみ） | ✅ 完了 |
| **Phase 2** | lint 強化（未知キー検出ほか）、容量ガードのオプトイン生成（案B）、ゴールデンテスト/自己テストのCI統合 | 提案3B,6 | 最小（`SqlRDBCount` ＋ `SQL_RDB_ERR_CAPACITY_EXCEEDED` を追加） | ✅ 完了 |
| **Phase 3（将来）** | PostgreSQL ドライバ、DB側トリガによる容量強制、容量上限の本体API化（必要時） | 提案3C,3D,7 | あり | 未着手 |

### Phase 2 で実際に加えた本体改修（最小・追加のみ）

案Bの容量ガードを効率的かつ正確（UPSERTを誤判定しない）に実装するには行数カウントが必要だが、公開APIに件数取得手段が無かったため、以下を**追加のみ**（既存シグネチャ・ABIは不変、列挙値は末尾に追加）で導入した:

- `SqlRDBResult SqlRDBCount(h, struct_name, cond, &out_count)` — 読み取り専用 `SELECT COUNT(*)`。`cond=NULL` で全件。
- `SQL_RDB_ERR_CAPACITY_EXCEEDED = -33` — 生成ガードが容量超過INSERTを拒否する際の戻り値。
- Query Builder に `C2SQL_QB_COUNT` を追加。

生成される `Write<Name>Guarded()` は `SqlRDBCount` の上に構築され、既存PKへのUPSERT（テーブルが増えない更新）は容量超過でも許可する。テストは `tests/test_capacity.c`、ツール自己テスト＋ゴールデン退行検知は `tests/test_tools.py`（ctest: `tools_pytests`）。

## 5. リスクと留意点

| リスク | 内容 | 緩和策 |
|--|--|--|
| 設計方針の転換 | `design.md` の「コード生成は非ゴール」と矛盾 | 「本体は非依存、生成は外部ツール」と再定義し本体の単純さは維持 |
| 仕様⇄構造体の乖離 | 生成後に利用者が構造体を手で書き換える | 構造体 `typedef` も生成し、利用者は触らない運用を推奨 |
| `max_records` の意味揺れ | 見積りか強制かが曖昧 | フォーマットに `enforce: true/false`（将来）を用意、既定は見積り |
| Postgres 移植性 | トリガ等バックエンド依存機能 | 容量保護は案B（アプリ層 `COUNT(*)`）を基本に、DB依存機能は限定 |
| TOCTOU（案B） | カウントとInsertの間に競合 | 同一ハンドルmutex内 or TX内で実施、厳密保証が要れば案D |

---

## 6. 次アクション（提案）

Phase 0〜2 は完了済み（本リポジトリに実装・テスト済み）。残りは Phase 3:

1. **PostgreSQL ドライバ**: 既存のドライバ抽象（vtable）に Postgres 実装を追加し、`backend: "postgres"` を解禁。
2. **DB側での容量強制**: SQLite トリガ／Postgres ルールによる `max_records` のDBレベル強制（案C）。バックエンド依存のため移植性とのトレードオフを評価。
3. **容量上限の本体API化（案D, 必要時）**: 生成ラッパではなく本体に容量メタを持たせる案。ニーズが固まってから。
4. **CI ワークフロー**: GitHub Actions で `tools/build_batch.sh` ＋ ASan/UBSan ＋ clang-tidy を回す（現状はローカルバッチ）。
